# 08. 测试标准

## 测试目标

本项目测试不是只证明代码能跑，而是证明系统不会误拨、不会泄露敏感信息、不会因为外设故障失控。

测试分层：

1. 单元测试。
2. 契约测试。
3. 配置测试。
4. fake 调度测试。
5. adapter 测试。
6. 硬件集成测试。
7. 微信 UI 人工验收。
8. 完整通话人工验收。

## 必跑测试

每次提交前必须运行：

```powershell
python -m pytest
python -m ruff check .
git diff --check
```

如果修改文档，也要运行这些命令，确保没有破坏项目。

## 单元测试

覆盖对象：

- 状态机。
- 配置解析。
- domain 对象。
- 文本预处理。
- prompt 加载。
- 错误分类。

标准：

- 不依赖真实手机。
- 不依赖声卡。
- 不依赖网络。
- 不依赖真实模型。
- 运行速度快。

示例：

- `IDLE + START_DIAL -> DIALING`
- 缺少 `audio.sample_rate` 抛出 `ConfigError`
- 空 TTS 文本拒绝合成
- Markdown 回复被清理

## 契约测试

覆盖对象：

- port。
- fake adapter。
- 占位 adapter。
- 真实 adapter 的输入输出结构。

标准：

- 每个 adapter 必须是对应 port 的实例。
- 返回对象必须是项目 dataclass。
- 不返回第三方 SDK 对象。
- 异常必须有清晰语义。

## 配置测试

覆盖对象：

- `config/config.example.yaml`
- prompt YAML。
- `.env` 加载。

标准：

- 示例配置永远可加载。
- 示例配置不包含真实秘密。
- 缺失字段失败。
- 错误类型失败。
- 路径字段解析正确。

建议后续测试：

- provider 枚举校验。
- `allow_real_call` 默认 false。
- 真实拨号模式必须显式开启。

## Fake 调度测试

覆盖对象：

- `Orchestrator`
- `CallSession`
- fake ports

标准：

- 正常 fake 一轮通话进入 `HUNG_UP`。
- 等待接通超时进入 `HUNG_UP`。
- ASR 空文本进入失败。
- LLM 空回复进入失败。
- TTS 报错进入失败。
- finally 会调用挂断和 stop。

## Adapter 测试

### 自动化 adapter

单元测试：

- fake uiautomator client。
- fake UI XML。
- fake 截图路径。

集成测试：

- 真实 ADB 设备。
- 截图。
- 打开微信。
- 搜索联系人。

禁止：

- 默认单元测试真实拨号。

### 音频 adapter

单元测试：

- fake stream。
- fake queue。
- fake playback。

集成测试：

- 枚举设备。
- 打开输入流。
- 打开输出流。
- 录制和播放测试音。

### TTS adapter

单元测试：

- fake HTTP server。
- 成功音频。
- 错误码。
- 超时。
- 空音频。

集成测试：

- 本地 GPT-SoVITS 服务。
- 短句合成。
- 音频解析。

### LLM adapter

单元测试：

- fake HTTP server。
- 401。
- 429。
- 500。
- 超时。
- 空文本。

集成测试：

- 真实 API 健康检查。
- Prompt 输出人工检查。

## 硬件集成测试

标记建议：

```python
@pytest.mark.integration
@pytest.mark.hardware
```

默认不进入普通测试集。

硬件测试项目：

- ADB 在线。
- 截图成功。
- 电量和充电状态可读取。
- 声卡输入输出可枚举。
- 输入输出流可打开。
- 静音底噪达标。
- 测试音不削波。

验收标准：

- 所有硬件测试输出报告。
- 硬件测试失败不能进入真实拨号。

## 微信 UI 人工验收

必须分级：

1. 设备截图。
2. 打开微信。
3. 搜索联系人。
4. 真实短拨号。
5. 完整通话。

每级验收记录：

- 日期。
- 手机型号。
- Android 版本。
- 微信版本。
- 屏幕分辨率。
- 配置文件摘要。
- 结果。
- 失败截图路径。

标准：

- 未通过前一级，不进入后一级。
- 联系人不唯一时禁止拨号。
- 默认配置禁止真实拨号。

## 完整通话人工验收

前置条件：

- ADB 通过。
- 音频通过。
- VAD 通过。
- ASR 通过。
- TTS 通过。
- LLM 通过。
- 微信 Level 4 通过。

测试步骤：

1. 人工在场。
2. B 机准备接听。
3. 执行 preflight。
4. 开启真实拨号。
5. 发起通话。
6. 接通后播放问候。
7. B 端说固定短句。
8. 系统回复一轮。
9. 系统挂断。
10. 检查日志。

验收标准：

- 不误拨。
- 音频可听清。
- 用户语音被正确切分。
- ASR 文本大致正确。
- LLM 回复短。
- TTS 可懂。
- 结束后微信不留在危险页面。

## 失败场景矩阵

必须覆盖：

- ADB 不存在。
- 无设备。
- 多设备。
- 手机未授权。
- 微信未登录。
- 联系人未找到。
- 联系人多个候选。
- 对方未接听。
- VAD 无语音。
- ASR 空文本。
- LLM 超时。
- TTS 服务未启动。
- 声卡设备丢失。
- Ctrl+C 中断。

每个失败场景标准：

- 不误拨。
- 不无限重试。
- 能挂断则挂断。
- 记录失败原因。
- 资源释放。

## 敏感信息测试

检查目标：

- 日志。
- 异常。
- 测试快照。
- 文档。
- Git diff。

禁止出现：

- API Key。
- 真实联系人姓名。
- 微信账号信息。
- 设备序列号。
- 私密音频文件。
- 模型权重。

建议命令：

```powershell
git diff --check
rg "api[_-]?key|secret|token|password" .
```

命令只是辅助，不能替代人工检查。
